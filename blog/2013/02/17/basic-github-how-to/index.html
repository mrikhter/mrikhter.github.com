Basic-github-how-to

Feb 17th, 2013

Github is an essential tool for working on multi-dev projects. Going through the prework for this program, github was one of the easier subjects to grasp…or so I thought. The first day of class would dramatically prove otherwise. Our first assignment was a deep dive into the mechanics of this collaborative tool and it was a schmess. With 28 people cloning, forking, commiting, pushing and merging I was left lost and confused. To be fair, the way in which we we’re working was not the most accurate depiction of normal workflow. In practice, it is rare to have multiple people working on the same file at the same time. Thus, I decided to create a basic workflow model to help beginners navigate their way through github with a decreased fear of overwriting work by people who actually know what they’re doing.

Are you the creator of the remote repository?
If so, create a remote repository on Github:

Next, create a local repository and point it to the remote repository. You need to obtain the url from the page that pops up after selecting Create repository:
$ git remote add url When running this command, make sure you are in the directory you want to be in.

Do you need access to an existing repository?
If so, clone to create a local copy of the remote repository. To clone, you need the url from github, which you can find on the repositoryâ€™s page.
$ git clone url name
Only include name if you want your local copy to have a different name than the remote.
